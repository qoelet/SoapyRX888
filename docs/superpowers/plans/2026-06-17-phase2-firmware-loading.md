# Phase 2: Make It Detect (FX3 Firmware Loading) — Implementation Plan

> **For agentic workers:** Steps use checkbox (`- [ ]`) syntax. Build is verified
> on local gcc + arm64 macOS CI; the actual USB upload is verified by the user on hardware.

**Goal:** When an RX888-family device is found in FX3 bootloader mode (`04b4:00f3`),
upload the bundled SDDC firmware so it re-enumerates as the RX888 (`04b4:00f1`) and
appears in `SoapySDRUtil --find`.

**Confirmed on the user's hardware (2026-06-17):** `SoapySDRUtil --find` reports
`addr=04b4:00f3 name=WestBridge module=FX3` — i.e. bare FX3 bootloader, no firmware.
This is precisely the state this phase resolves.

**Architecture:** Firmware embedded as a C array in `librx888`. A new
`rx888_load_firmware()` scans for `04b4:00f3`, uploads the CYFX2 RAM image via the
Cypress vendor protocol (request `0xA0`), jumps to the entry point, and waits for
re-enumeration. The SoapySDR enumerator calls it before counting devices. Because a
flashed device becomes `00f1`, the call is naturally idempotent.

---

### Task 1 — Bundle firmware (DONE)
- `librx888/firmware/sddc_fx3_firmware.h` — auto-generated byte array of `SDDC_FX3.img`.
- `librx888/firmware/PROVENANCE.md` — source URL, SHA-256
  `c1adb233b327f45c5d40c8f3847e3c31030bcbb7eed0e98893c667e2ec3dadaa`, GPLv3.
- Verified: CYFX2 image, `bImageType=0xB0`, 4 sections, entry point `0x40012cfc`,
  146188 bytes fully consumed.

### Task 2 — FX3 RAM downloader (DONE)
- `librx888/src/librx888.c`: `static int fx3_ram_download()` parses CYFX2 sections,
  writes each in ≤2048-byte control transfers (`0xA0`, wValue/wIndex = address), then
  issues a zero-length transfer to the entry point to start firmware.

### Task 3 — Public loader + re-enumeration wait (DONE)
- `librx888/src/librx888.c`: `int rx888_load_firmware(void)` scans `04b4:00f3`,
  best-effort claims interface 0 (EP0 control transfers don't require it), uploads,
  then polls `rx888_get_device_count()` up to ~5 s for the `00f1` device.
- Declared in `librx888/include/librx888.h`.

### Task 4 — Build wiring (DONE)
- `librx888/src/CMakeLists.txt`: add `firmware/` to `rx888` PRIVATE include dirs.

### Task 5 — Call from SoapySDR enumerator (DONE)
- `SoapyRX888/Registration.cpp`: `findRX888()` calls `rx888_load_firmware()` before
  `rx888_get_device_count()`.

### Task 6 — Verify
- [x] librx888 builds clean on gcc; `rx888_load_firmware` exported.
- [ ] arm64 macOS CI green (compiles `Registration.cpp` against new librx888).
- [ ] **User hardware test:** rebuild/install on Mac, then `SoapySDRUtil --find`
      should now show `driver=rx888` / `04b4:00f1` after auto-flash (watch stderr for
      "FX3 in bootloader mode, uploading firmware..."). USB-3 (blue/SS) port + cable
      recommended so the re-enumerated device runs at SuperSpeed.

## Phase 2 exit criteria
Plug in → `SoapySDRUtil --find` lists the RX888 as an `rx888` device (firmware
auto-loaded), no manual steps.

## Risk notes
- If macOS blocks claiming the FX3, the loader proceeds without a claimed interface
  (EP0 transfers usually still work); if upload fails, the stderr line names the
  failing address.
- If the device re-enumerates but `lime` keeps claiming `00f1`: it won't — SoapyLMS
  matches the bootloader `00f3`, not the loaded `00f1`.
